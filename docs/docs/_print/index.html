The cloud native cluster optimization platform (Cluster Optimizer) provides cloud native cloud native cluster optimization capabilities.

Cluster Optimizer can help users optimize cloud native clusters. Its main functions include:

  • Provides a variety of optimization strategies, including optimization suggestions from multiple dimensions such as node groups, nodes, disks, persistence and applications.
  • Cloud account (access credentials) management and cluster management.
  • User management.

1 - Installation

Describes the detailed steps for installing the cloud-native cluster optimization platform (Cluster Optimizer).

1.1 - Install via kubectl

Describes the detailed steps to install the cloud-native cluster optimization platform (Cluster Optimizer) through kubectl.

This page describes the detailed steps to install the one-stop cloud native cluster optimization platform (Cluster Optimizer).

Preconditions

  • Mysql database, version 5.7 and above
  • Kubernetes cluster, version 1.22 and above
  • The certificate of the Kubernetes cluster can create the wiseinf-system namespace and create Deployment, ConfigMap, Service and other resources under this namespace

Clone installation script

  1. Open the command line console and clone the installation script to the local computer through the following command.

        git clone https://github.com/wiseinf/setup.git

    1. If you use the Gitee warehouse address, you need to replace https://github.com/wiseinf/setup.git in the above command with https://gitee.com/wiseinf/setup.git.

    2. The subsequent steps assume that the setup library has been cloned to the user’s home directory. If it has not been cloned to the user’s home directory, the file path in the command needs to be adjusted according to the actual situation.

Deploy application

  1. Create the namespace wiseinf-system through the following command:

        kubectl apply -f ~/setup/cluster-optimizer/yaml/namespace.yaml

  2. Create the database optimizer-mysql through the following command, its service name is optimizer-mysql, and can be accessed through the user root and the password Optimizer~:

        kubectl apply -f ~/setup/cluster-optimizer/yaml/mysql.yaml

    If you need to bind the database to the specified node, please edit the ~/setup/cluster-optimizer/yaml/mysql.yaml file, remove the comments in the affinity field, and modify the value of the value field to Specify the node name as shown below.

      ...
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/hostname
            operator: In
            values:
            - ip-10-16-121-100.us-west-1.compute.internal
  ...

  3. Create Deployment optimizer through the following command:

        kubectl apply -f ~/setup/cluster-optimizer/yaml/optimizer.yaml

    Note: If you want to use a deployed database, please see [Use Deployed Database](#Use Deployed Database).

  4. Check whether optimizer is running normally through the following command:

        kubectl get pods -n wiseinf-system

    If it runs normally, you can get output similar to the following, in which the currently running Pod Name is optimizer-5b956655df-gspq6:

        NAME READY STATUS RESTARTS AGE
    optimizer-5b956655df-gspq6 2/2 Running 0 1m

  5. Create port-forward to access the optimizer service through the following command:

        kubectl port-forward svc/optimizer 8000:80 -n wiseinf-system

  6. Open the browser and visit the address http://localhost:8000/.

  7. If you can see the Cluster Optimizer homepage, the application Cluster Optimizer is deployed successfully.

Use deployed database

If you want to use the deployed database and use a new username and password to access Cluster Optimizer, please refer to the following steps:

  1. Execute the mysql command on the command line console to log in to the Mysql database. 127.0.0.1 is the Mysql database host address. You need to replace it with the actual address you use.

        mysql -h127.0.0.1 -uroot

    Note: If you use a remote database, make sure the database is accessible from the Kubernetes cluster.

  2. Create database schema optconf (optimization configuration library) and optdata (optimization database).

        mysql> source ~/setup/cluster-optimizer/sql/create_database.sql

    Note: Please be aware of the risk of data loss. If the optconf library and optdata library already exist in the mysql database, the above script will first delete the library and then create a new library.

  3. Create a new username and password, and set corresponding permissions for them. The username in the script is optimizer and the password is Optimizer~. You can adjust it according to your needs.

        mysql> source ~/setup/cluster-optimizer/sql/create_user.sql

  4. Adjust the database information in the Cluster Optimizer configuration file. Edit the ConfigMap optimizer-conf in the file ~/setup/cluster-optimizer/yaml/optimizer.yaml, and change the dataSource.name value under the configuration items optdata and optconf The user name, user password, database address and port are adjusted to the corresponding information of the Mysql database.

  5. Restart the application and check the log output of Cluster Optimizer (as shown below) to see whether the database configuration information is consistent with your expectations.

    {
    ...
    "optconf": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optconf?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",
            "maxOpenConns": 10
        }
    },
    "optdata": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optdata?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",
            "maxOpenConns": 10
        }
    },
    ...
}

    At the same time, also check whether there are any database-related errors. If there are no related errors, it proves that a new database has been used.

Next step: Import the cluster

Next, you can add a cluster and view its related optimization recommendations. For details, see:

  • Import AWS EKS cluster
  • Import Alibaba Cloud Cluster

1.2 - Install via Helm

Describes the detailed steps to install the cloud-native cluster optimization platform (Cluster Optimizer) through Helm.

This page describes the detailed steps for installing Cluster Optimizer via Helm.

Preconditions

  • Mysql database, version 5.7 and above
  • Kubernetes cluster, version 1.22 and above
  • The certificate of the Kubernetes cluster can create the wiseinf-system namespace and create Deployment, ConfigMap, Service and other resources under this namespace
  • Helm, 3.0.0 and above recommended

Deploy application

  1. Add the Helm warehouse cluster-optimizer through the following command:

        helm repo add cluster-optimizer https://wiseinf.github.io/cluster-optimizer/

  2. Update the warehouse through the following command:

        helm repo update

  3. View the Charts contained in the warehouse cluster-optimizer through the following command:

        helm search repo cluster-optimizer

    The output is as follows:

    NAME CHART VERSION APP VERSION DESCRIPTION
cluster-optimizer/cluster-optimizer 0.1.0 1.1.0 A Helm chart for cluster optimizer

    It can be seen that the warehouse includes a Chart named cluster-optimizer/cluster-optimizer.

  4. Install the demo Chart through the following command:

        helm install demo cluster-optimizer/cluster-optimizer -n wiseinf-system --create-namespace

    Among them, -n wiseinf-system means to install demo under the namespace wiseinf-system, and the parameter --create-namespace means to create this namespace. The output is as follows:

    NAME: demo
LAST DEPLOYED: Thu Mar 28 15:54:42 2024
NAMESPACE: default
STATUS: deployed
REVISION: 1
...

    It means that Helm has successfully deployed the demo Chart.

  5. Check whether optimizer is running normally through the following command:

        kubectl get pods -n wiseinf-system

    The output is as follows:

    NAME READY STATUS RESTARTS AGE
cluster-optimizer-demo-7bfdfdf954-2cgqb 2/2 Running 0 108s
cluster-optimizer-demo-optimizerdb-5d6cdb6b9f-94lc7 1/1 Running 0 108s

    As you can see from the output, Helm has deployed two Pods, of which cluster-optimizer-demo-7bfdfdf954-2cgqb is the Pod of Cluster Optimizer, and cluster-optimizer-demo-optimizerdb-5d6cdb6b9f-94lc7 is the Mysql database it depends on. .

    Note: If you want to use a deployed database, please see [Use Deployed Database](#Use Deployed Database).

  6. Create port-forward to access the optimizer service through the following command:

        kubectl port-forward svc/optimizer 8000:80 -n wiseinf-system

  7. Open the browser and visit the address http://localhost:8000/.

  8. If you can see the Cluster Optimizer homepage, the application Cluster Optimizer is deployed successfully.

  9. You can delete the demo Chart through the command:

    helm uninstall demo

Use deployed database

If you want to use the deployed database and use a new username and password to access Cluster Optimizer, please refer to the following steps:

  1. Open the command line console and clone the installation script to the local computer through the following command.

        git clone https://github.com/wiseinf/setup.git

    1. If you use the Gitee warehouse address, you need to replace https://github.com/wiseinf/setup.git in the above command with https://gitee.com/wiseinf/setup.git.

    2. The subsequent steps assume that the setup library has been cloned to the user’s home directory. If it has not been cloned to the user’s home directory, the file path in the command needs to be adjusted according to the actual situation.

  2. Execute the mysql command on the command line console to log in to the Mysql database. 127.0.0.1 is the Mysql database host address. You need to replace it with the actual address you use.

        mysql -h127.0.0.1 -uroot

    Note: If you use a remote database, make sure the database is accessible from the Kubernetes cluster.

  3. Create database schema optconf (optimization configuration library) and optdata (optimization database).

        mysql> source ~/setup/cluster-optimizer/sql/create_database.sql

    Note: Please be aware of the risk of data loss. If the optconf library and optdata library already exist in the mysql database, the above script will first delete the library and then create a new library.

  4. Create a new username and password, and set appropriate permissions for them. The username in the script is optimizer and the password is Optimizer~. You can adjust it according to your needs.

        mysql> source ~/setup/cluster-optimizer/sql/create_user.sql

  5. Create the file values.yaml and edit its content as follows:

    optimizedb:
  conf:
    dbHost: 127.0.0.1
    schema:optconf
    port: "3306"
    user: optimizer
    password: Optimizer~
  data:
    dbHost: 127.0.0.1
    schema: optdata
    port: "3306"
    user: optimizer
    password: Optimizer~

    Among them, dbHost is the database address, schema is the database Schema, port is the port, user and password are the username and password respectively.

  6. Install the demo Chart through the following command:

        helm install demo cluster-optimizer/cluster-optimizer -f values.yaml --namespace wiseinf-system

    You can also adjust the database settings by upgrading the original deployment:

        helm upgrade demo cluster-optimizer/cluster-optimizer -f values.yaml --namespace wiseinf-system

  7. Restart the application, check the log output of the Cluster Optimizer Pod (as shown below), and check whether the database configuration information is consistent with your expectations.

    {
    ...
    "optconf": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optconf?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",
            "maxOpenConns": 10
        }
    },
    "optdata": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optdata?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",

  • The community version of the cluster optimization platform has been deployed. For specific installation steps, please refer to [Installation via Kubectl] (/docs/setup/setup-using-kubectl/) and [Installation via Helm] (/docs/setup/setup-using-helm/).

  1. Open the command line console and clone the installation script to the local computer through the following command.

        git clone https://github.com/wiseinf/setup.git

    user:
  dataSource:
    name: "root:optimizer@tcp(optimizer-mysql:3306)/user?createDatabaseIfNotExist=true&charset=utf8mb4&parseTime=true&loc=Local"
    maxOpenConns: 10

Configure commercial license

  1. Adjust the database information in the Cluster Optimizer configuration file. Edit ConfigMapoptimizer-conf in the file ~/setup/cluster-optimizer/yaml/optimizer.yaml and add the following configuration items.

    license:
  certificate: {obtained authorization certificate}
  signature: {obtained authorized signature}

  2. Deploy Deployment optimizer through the following command:

        kubectl apply -f ~/setup/cluster-optimizer/yaml/optimizer.yaml

    Note:

    1. If the Pod of Deployment optimizer has not been restarted, please use kubectl delete pod -l app=optimizer -n wiseinf-system to delete the Pod of Deployment optimizer and then re-create it. `

System initialization

  2. Visit the address http://localhost:8000/user/login, enter the user name and password to log in to the system.

  3. At this point, you have upgraded the system to the commercial version.

Describes the detailed steps to import an Alibaba Cloud cluster.

This section describes the detailed steps for importing Alibaba Cloud clusters.

Preconditions

  1. Obtain the Alibaba Cloud Access Key and Access Key Secret and have the following permissions. For the specific application process, please refer to Create Access Key.

    Services Permissions Description

For details, please refer to [Creating a custom permission policy through script editing mode](https://help.aliyun.com/zh/ram/user- guide/create-a-custom-policy#section-kwn-gu8-48m)

Add Alibaba Cloud cluster

It also provides an operation bar to facilitate users to perform corresponding operations. As shown below.

  2. Click the Import Cluster button to pop up the import dialog box, where you can import the cluster. There are 5 steps to import a cluster. Currently, only Alibaba Cloud clusters are supported.

    1. Set Alibaba Cloud access credentials
    2. Connect to the Kubernetes cluster
    3. Connect to Prometheus
    4. View configuration
    5. Complete

  3. The interface for setting Alibaba Cloud access credentials is as shown in the figure below. In the Alibaba Cloud access credentials interface, set the Alibaba Cloud access credentials by using existing credentials. Here, select the Alibaba Cloud account added in Add Alibaba Cloud Account, as shown in the figure below.

  4. After setting the Alibaba Cloud access credentials, click the Next button to enter the page for connecting to the Kubernetes cluster, as shown in the figure below. The platform supports connecting to two types of Kubernetes clusters, including managed clusters and self-built clusters. The Connect Hosting Cluster page is shown in the figure below. Here we choose the managed cluster. The platform automatically obtains the user cluster through the cloud account. The user selects the corresponding cluster through the cluster drop-down box, enters the description field, selects whether to use the API Server private network address, selects whether to automatically refresh KubeConfig and the KubeConfig life cycle. Finally enter the node group label.

  5. The connection page to Prometheus is as shown in the figure below. After the user enters information on this page, click the Next button to enter the view configuration page. The input information details are shown in the table below.

    The parameter description for connecting to Prometheus is as shown in the table:

    Parameters Description
    Whether Thanos If the backend cluster is Thanos, this should be checked.
    Prometheus address Prometheus address, must be accessible.
    Limit access rate If you need the current platform’s access rate to Prometheus, you should check this option.
    Maximum number of concurrencies The maximum number of concurrencies for the platform to access Prometheus.
    Cluster label Cluster label is used to filter the monitoring indicators of the cluster from Prometheus or Thanos
    Cluster tag value Cluster tag value is used to filter the monitoring indicators of the cluster from Prometheus or Thanos

  6. View the configuration page as shown in the figure below. This page displays Alibaba Cloud access credentials, detailed parameters for connecting to the Kubernetes cluster and connecting to the Prometheus cluster, which is convenient for users to confirm. When the user completes the confirmation, click the Import button to enter the completion page.

  7. The completed page is as shown below, and the submission status will be displayed on the page. Click Finish to complete the cluster import.

Next steps

After importing the cluster, wait for a period of time (waiting for the optimization strategy to complete scheduling and execution), and then you can view optimization suggestions for the cluster in multiple dimensions, including:

  • Nodegroup
  • Node
  • GPU node
  • Disk
  • Persistent volume
  • Application

Describes detailed steps to import an AWS EKS cluster.

This section describes the detailed steps to import an AWS EKS cluster.

Preconditions

    {

      "Action": [
        "ec2:DescribeInstances",

Import Kuberentes cluster

  • Set cluster hibernation *Hibernate immediately

Set cluster hibernation

Cluster hibernation reduces cluster resource usage and saves costs by properly managing the nodes in the cluster. Assume that the cluster starts sleeping at 9 pm from Monday to Friday and wakes up at 8 am from Monday to Friday (that is, ending hibernation). The machine cost savings can be as high as 54.8%. Setting cluster hibernation allows you to set the sleep time and wakeup time of the cluster. During the sleep time, the status of the load in the cluster is stored, and the node resources in the cluster are gradually released. During the wake-up time, the system gradually restores the node resources in the cluster and restores the original state of the load. Cluster hibernation is especially suitable for offline environments such as development and testing.

Click Set Cluster Hibernation in the operation bar, and the dialog box will pop up as shown below:

Select Enable in the Whether to enable sleep option, then set the specific sleep time and wake up time, and click the OK button. The default sleep time is 9pm from Monday to Friday, and the default wake-up time is 8am from Monday to Friday.

If you want to cancel cluster hibernation, select Disable in the Whether to enable hibernation option and click the OK button.

When the cluster is dormant, the cluster access account provided by the user will be used to operate the cluster. You need to set up a cluster access account. For details, please see Setting up a cluster access account.

Note: The interval between sleep time and wake up time must be greater than 2 hours.

Hibernate now

Wake up immediately

Move the mouse to ... and click the Wake Up Now button to wake up the cluster immediately.

log

3.2 - Configuration parameters

Describes the configuration parameters of the cloud native cluster optimization platform configuration file.

Describes the configuration parameters of the cloud native cluster optimization platform configuration file.

Overview

The cloud-native cluster optimization platform must rely on configuration files to run. The configuration file location can be specified through the command line parameter --config, for example: --config=conf/cluster-optimizer.yaml. The default path is conf/cluster-optimizer.yaml

The configuration file format is YAML format, the example is as follows:

server:
  port: 7070
engine:
  maxQueryDuration: 4h
optconf:
  dataSource:
    name: "root:optimizer@tcp(127.0.0.1:3306)/optconf?createDatabaseIfNotExist=true&charset=utf8mb4&readTimeout=30s&writeTimeout=30s&parseTime=true&loc=Local"
    maxOpenConns: 10
optdata:
  dataSource:
    name: "root:optimizer@tcp(127.0.0.1:3306)/optdata?createDatabaseIfNotExist=true&charset=utf8mb4&readTimeout=30s&writeTimeout=30s&parseTime=true&loc=Local"
    maxOpenConns: 10
optimize:
  defaultCron: "0 0 1 * * ?"
  strategyConfigs:
    "OOM":
      enabled: false
    "Resource Recommendation":
      enabled: false
    "Low Utilization Node":
      enabled: false
    "No Quota":
      enabled: false
  enabled: true
sms:
  enabled: false
  ...
notification:
  templateBasePath: templates/notification

Configuration parameters

server.port

The parameter server.port is the external service port of the cloud native cluster optimization platform. The default value is 7070.

engine.maxQueryDuration

The parameter engine.maxQueryDuration sets the maximum query period of the query measurement server (such as Prometheus). The default value is 1 day. When the query time range exceeds the maximum query period, it will be divided into multiple query windows for separate queries, thereby reducing the pressure on the server.

optconf

The parameter optconf is used to configure the optimization configuration library.

Its sub-parameters include dataSource.name and maxOpenConns, which are explained below.

dataSource.name

The parameter dataSource.name is used to set the Mysql data source name, for example root:optimizer@tcp(127.0.0.1:3306)/optconf?createDatabaseIfNotExist=true&charset=utf8mb4&readTimeout=30s&writeTimeout=30s&parseTime=true&loc=Local. Among them, createDatabaseIfNotExist means that when the database does not exist, create this database first.

maxOpenConns

The parameter maxOpenConns sets the maximum number of open connections to the database.

optdata

The parameter optdata is used to configure the optimization database.

Its sub-parameters include dataSource.name and maxOpenConns, whose meanings are consistent with the corresponding sub-parameters of optconf and will not be explained again.

optimize

The parameter optimize is used to set parameters related to the optimization engine. Its sub-parameter settings will be described below.

defaultCron

The parameter defaultCron is used to set the default scheduling parameters of the optimization engine. The default value is 0 0 1 * * ? (executed at 1 am every day),

enabled

The parameter enabled sets whether to enable the optimization engine. The default value is true.

strategyConfigs

The parameter strategyConfigs is used to configure the strategy configuration of the optimization engine. It is configured as a key-value pair. The key is the name of the policy and the value is the configuration of the policy.

The policy name is a string, and the content of the policy value is as follows:

  enabled: false
  cron: "0 0 1 * * ?"
  parameters:
    - key1: value1

4.1 - Periodic notifications of optimization suggestions

Describes how to set up a periodic optimization suggestion notification mechanism through Cluster Optimizer to facilitate obtaining corresponding optimization suggestions via email.

Describes how to set up a periodic optimization suggestion notification mechanism through Cluster Optimizer to facilitate obtaining corresponding optimization suggestions via email.

Overview

During the operation of Kubernetes clusters, we often need to obtain key indicators or information regularly and communicate this information to relevant users through emails or other notification methods. The periodic notification mechanism allows users to make necessary adjustments and optimization operations based on the information received.

The cluster optimization platform can automatically monitor possible problems and potential optimization points in the cluster, such as low node utilization, application memory overflow, unused Persistent Volume, etc., and generate optimization suggestions containing detailed information and recommended parameters, and notify users via email. . For example, when an application memory overflow is detected, the system will automatically collect relevant information and send this information to the person in charge or responsible person of the application. This not only improves the efficiency of problem response, but also ensures efficient utilization of cluster resources and stable operation of applications. In this way, users can more proactively manage and maintain clusters and applications in the cluster, improving overall operational efficiency.

Usage scenarios

Users want to obtain common optimization advice information through emails, such as low node utilization, application memory overflow, Persistent Volume is not used, etc., and perform corresponding operations based on these optimization advice information. This article will take configuring node group optimization recommendations as an example to outline how to configure Cluster Optimizer so that it can provide periodic optimization recommendation notifications.

Suppose we expect to notify user Jay (email address is jay@wiseinf.com) via email at 8 a.m. every day of the node group optimization suggestions in the cluster. The specific preconditions and configuration process are as follows.

Assume that the user has the following needs:

  1. Send the node group optimization suggestion results to user Jay (the email address is jay@wiseinf.com)
  2. Receive notifications via email at 8 a.m. daily

The cluster optimization platform provides notifications to users through the following two components:

  • Notification channel: Provides basic configuration parameters for sending notifications. For example, for email notifications, typical configurations include host address, port, user name and password, etc. Notification channels can be used by multiple notification rules.
  • Notification rules: Provide configuration parameters, recipient addresses, etc. for constructing notification content.

The following describes how to configure notification channels and notification rules through the cluster optimization platform so that it can meet the above user needs.

Cloud service provider related faults

Amazon AWS

  1. Failed to obtain AWS credentials, the log shows getting credentials: exec: executable aws failed with exit code 253.

    The main reason: Use the aws command to obtain credentials when accessing EKS Kubernetes in AWS. aws is not installed by default in Cluster Optimizer. It obtains credentials through the API.
