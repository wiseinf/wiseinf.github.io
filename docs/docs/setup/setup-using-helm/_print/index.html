Install via Helm

Describes the detailed steps to install the cloud-native cluster optimization platform (Cluster Optimizer) through Helm.

This page describes the detailed steps for installing Cluster Optimizer via Helm.

Preconditions

  • Mysql database, version 5.7 and above
  • Kubernetes cluster, version 1.22 and above
  • The certificate of the Kubernetes cluster can create the wiseinf-system namespace and create Deployment, ConfigMap, Service and other resources under this namespace
  • Helm, 3.0.0 and above recommended

Deploy application

  1. Add the Helm warehouse cluster-optimizer through the following command:

        helm repo add cluster-optimizer https://wiseinf.github.io/cluster-optimizer/

  2. Update the warehouse through the following command:

        helm repo update

  3. View the Charts contained in the warehouse cluster-optimizer through the following command:

        helm search repo cluster-optimizer

    The output is as follows:

    NAME CHART VERSION APP VERSION DESCRIPTION
cluster-optimizer/cluster-optimizer 0.1.0 1.1.0 A Helm chart for cluster optimizer

    It can be seen that the warehouse includes a Chart named cluster-optimizer/cluster-optimizer.

  4. Install the demo Chart through the following command:

        helm install demo cluster-optimizer/cluster-optimizer -n wiseinf-system --create-namespace

    Among them, -n wiseinf-system means to install demo under the namespace wiseinf-system, and the parameter --create-namespace means to create this namespace. The output is as follows:

    NAME: demo
LAST DEPLOYED: Thu Mar 28 15:54:42 2024
NAMESPACE: default
STATUS: deployed
REVISION: 1
...

    It means that Helm has successfully deployed the demo Chart.

  5. Check whether optimizer is running normally through the following command:

        kubectl get pods -n wiseinf-system

    The output is as follows:

    NAME READY STATUS RESTARTS AGE
cluster-optimizer-demo-7bfdfdf954-2cgqb 2/2 Running 0 108s
cluster-optimizer-demo-optimizerdb-5d6cdb6b9f-94lc7 1/1 Running 0 108s

    As you can see from the output, Helm has deployed two Pods, of which cluster-optimizer-demo-7bfdfdf954-2cgqb is the Pod of Cluster Optimizer, and cluster-optimizer-demo-optimizerdb-5d6cdb6b9f-94lc7 is the Mysql database it depends on. .

    Note: If you want to use a deployed database, please see [Use Deployed Database](#Use Deployed Database).

  6. Create port-forward to access the optimizer service through the following command:

        kubectl port-forward svc/optimizer 8000:80 -n wiseinf-system

  7. Open the browser and visit the address http://localhost:8000/.

  8. If you can see the Cluster Optimizer homepage, the application Cluster Optimizer is deployed successfully.

  9. You can delete the demo Chart through the command:

    helm uninstall demo

Use deployed database

If you want to use the deployed database and use a new username and password to access Cluster Optimizer, please refer to the following steps:

  1. Open the command line console and clone the installation script to the local computer through the following command.

        git clone https://github.com/wiseinf/setup.git

    1. If you use the Gitee warehouse address, you need to replace https://github.com/wiseinf/setup.git in the above command with https://gitee.com/wiseinf/setup.git.

    2. The subsequent steps assume that the setup library has been cloned to the user’s home directory. If it has not been cloned to the user’s home directory, the file path in the command needs to be adjusted according to the actual situation.

  2. Execute the mysql command on the command line console to log in to the Mysql database. 127.0.0.1 is the Mysql database host address. You need to replace it with the actual address you use.

        mysql -h127.0.0.1 -uroot

    Note: If you use a remote database, make sure the database is accessible from the Kubernetes cluster.

  3. Create database schema optconf (optimization configuration library) and optdata (optimization database).

        mysql> source ~/setup/cluster-optimizer/sql/create_database.sql

    Note: Please be aware of the risk of data loss. If the optconf library and optdata library already exist in the mysql database, the above script will first delete the library and then create a new library.

  4. Create a new username and password, and set appropriate permissions for them. The username in the script is optimizer and the password is Optimizer~. You can adjust it according to your needs.

        mysql> source ~/setup/cluster-optimizer/sql/create_user.sql

  5. Create the file values.yaml and edit its content as follows:

    optimizedb:
  conf:
    dbHost: 127.0.0.1
    schema:optconf
    port: "3306"
    user: optimizer
    password: Optimizer~
  data:
    dbHost: 127.0.0.1
    schema: optdata
    port: "3306"
    user: optimizer
    password: Optimizer~

    Among them, dbHost is the database address, schema is the database Schema, port is the port, user and password are the username and password respectively.

  6. Install the demo Chart through the following command:

        helm install demo cluster-optimizer/cluster-optimizer -f values.yaml --namespace wiseinf-system

    You can also adjust the database settings by upgrading the original deployment:

        helm upgrade demo cluster-optimizer/cluster-optimizer -f values.yaml --namespace wiseinf-system

  7. Restart the application, check the log output of the Cluster Optimizer Pod (as shown below), and check whether the database configuration information is consistent with your expectations.

    {
    ...
    "optconf": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optconf?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",
            "maxOpenConns": 10
        }
    },
    "optdata": {
        "dataSource": {
            "name": "optimizer:Optimizer~@tcp(127.0.0.1:3306)/optdata?createDatabaseIfNotExist=true\u0026charset=utf8mb4\u0026parseTime=true\u0026loc=Local",
            "maxOpenConns": 10
        }
    },
    ...
}

    At the same time, also check whether there are any database-related errors. If there are no related errors, it proves that a new database has been used.
